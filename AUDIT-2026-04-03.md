# OpinAI Controller — Full Codebase Audit Report

**Date:** 2026-04-03
**Auditor:** Claude Code
**Scope:** All Go source, templates, Dockerfile, go.mod

---

## Executive Summary — Top 5 Most Critical Issues

1. **CRITICAL: Race condition on `JobManager.recorded` map** — concurrent goroutine access (watcher, harvester, API callbacks) with no synchronization. Will cause `concurrent map read and map write` panic under load.

2. **CRITICAL: XSS vulnerability in `/api/report/{id}` HTML output** — `run.Report` is inserted into HTML without escaping. An attacker can inject `<script>` tags via crafted issue content that flows through AI analysis into the report.

3. **CRITICAL: No authentication on any API endpoint** — all endpoints (admin, reproduction triggers, internal result injection, settings modification) are publicly accessible. Combined with permissive CORS (reflects any Origin), any website can trigger actions.

4. **CRITICAL: Race condition on rate limiter buckets** — `rateBucket.allow()` is called outside the store's mutex, so concurrent requests from the same IP race on `tokens` and `lastFill` fields.

5. **HIGH: `CreateVerifyFixJob` uses process-wide env vars for state passing** — `os.Setenv("_OPINAI_VERIFY_FIX_PENDING", "true")` is a process-global side-effect. Concurrent verify-fix requests will race.

## Issues by Severity

### CRITICAL

#### C1: Race condition on `JobManager.recorded` map
**File:** `internal/controller/jobs.go:39,332-337,345-348,353-357,362-363,436`
**Description:** The `recorded` map is a plain `map[string]bool` accessed from:
- `StartWatcher()` goroutine (`runWatch` → line 332)
- `HarvestCompletedJobs()` (called from poller goroutine → line 362)
- `MarkRecorded()` (called from HTTP handler goroutine → line 355)
- `ClearRecorded()` (called from HTTP handler goroutine → line 347)

No mutex protects this map. Under concurrent access, Go's runtime will panic with `concurrent map read and map write`.

**Fix:** Add a `sync.Mutex` to `JobManager` and lock around all `recorded` map accesses.

#### C2: XSS in `/api/report/{id}` HTML response
**File:** `internal/dashboard/features.go:89`
**Description:** `fmt.Fprintf(w, "<div>%s</div>", run.Report)` inserts report content directly into HTML without escaping. Report content originates from AI analysis of user-submitted GitHub issues, which could contain malicious HTML/JS.

**Fix:** Use `html.EscapeString()` on the report content before inserting into HTML.

#### C3: No authentication on API endpoints
**File:** `internal/dashboard/server.go:240-334`
**Description:** Every endpoint is publicly accessible:
- `/api/internal/result` — allows injecting fake reproduction results
- `/api/admin/*` — allows modifying settings, deleting repos, triggering analysis
- `/api/reproduce`, `/api/rerun/*` — allows triggering K8s Job creation
- `/api/webhook/github` — webhook endpoint (has optional HMAC verification)

Combined with CORS that reflects any origin (`middleware.go:60`), any website can make authenticated requests.

**Suggested fix:** Add at minimum a shared-secret bearer token for admin/internal endpoints. Document as a known limitation for the current deployment model (internal cluster network).

#### C4: Race condition on rate limiter buckets
**File:** `internal/dashboard/middleware.go:82-95,127-136,150-181`
**Description:** `getBucket()` acquires the store mutex to find/create the bucket, then releases it. The caller then calls `bucket.allow()` without any lock. Two concurrent requests from the same IP will race on the `tokens` and `lastFill` fields of the same `rateBucket`.

**Fix:** Move `allow()` inside the store's lock, or add per-bucket synchronization.

#### C5: Race condition in `CreateVerifyFixJob`
**File:** `internal/controller/jobs.go:296-297`
**Description:** `os.Setenv("_OPINAI_VERIFY_FIX_PENDING", "true")` sets a process-global environment variable. If two verify-fix requests arrive concurrently, the second `Setenv` will affect the first's `CreateReproductionJob` call (or vice versa with `Unsetenv`).

**Fix:** Pass the verify-fix flag as a parameter to `CreateReproductionJob` instead of using env vars.

#### C6: Race condition on `_sseK8sClient` global
**File:** `internal/dashboard/sse.go:566-586`
**Description:** `_sseK8sClient` is lazily initialized without synchronization. Two concurrent SSE requests could both see `nil` and both attempt to initialize, causing a race.

**Fix:** Use `sync.Once` for initialization.

### HIGH

#### H1: Double "done" SSE event in `handleChatStream`
**File:** `internal/dashboard/sse.go:391,399`
**Description:** Both the streaming and fallback code paths write a `done` event, then line 399 writes another `done` event unconditionally. Clients may misinterpret the stream.

**Fix:** Remove the duplicate `done` event on line 399 (move it inside the `else` block or restructure).

#### H2: Database write mutex bypassed in `handleRerunAll` and webhook handler
**File:** `internal/dashboard/features.go:34,183`
**Description:** Direct `database.DB().Exec(...)` calls bypass the `mu.Lock()` that all other write functions in the database package use. This can cause SQLite `SQLITE_BUSY` errors or corruption under concurrent writes.

**Fix:** Add a `DeleteProcessedIssues(repo, issue)` function to the database package that properly acquires the mutex.

#### H3: `DeleteIssueRuns` silently ignores errors
**File:** `internal/database/sqlite.go:290-293`
**Description:** Both `db.Exec` calls ignore their error returns, and the function always returns `nil`. If the DELETE fails (e.g., disk full), callers won't know.

**Fix:** Check and return errors.

#### H4: `GetStats` and `GetTotalStats` ignore QueryRow errors
**File:** `internal/database/sqlite.go:248-253,261-266`
**Description:** `db.QueryRow(...).Scan(...)` errors are silently discarded. If the query fails, zero values are returned without indication.

**Fix:** Check errors (at minimum log warnings).

#### H5: `handlePostComment` uses `http.DefaultClient` (no timeout)
**File:** `internal/dashboard/api.go:194`
**Description:** `http.DefaultClient.Do(req2)` has no timeout. If GitHub hangs, this request handler blocks forever, consuming a goroutine.

**Fix:** Use a client with a timeout.

#### H6: No timeout on git clone commands
**File:** `internal/runner/runner.go:422`, `internal/dashboard/sse.go:239`, `internal/sandbox/manager.go:552`
**Description:** `exec.Command("git", "clone", ...)` runs without a context timeout. A slow or hanging clone blocks the goroutine indefinitely.

**Fix:** Use `exec.CommandContext` with a timeout.

#### H7: `handleReproduceStream` can match stale runs
**File:** `internal/dashboard/sse.go:292-299`
**Description:** The polling loop queries `GetRuns(repo, 5)` and checks if any run matches the issue number. This will match a PREVIOUS run for the same issue, causing the SSE stream to immediately report "done" even though the new job hasn't started yet.

**Fix:** Record the timestamp before creating the job and only match runs created after that timestamp.

#### H8: `handleRunHistory` fetches all runs then filters in Go
**File:** `internal/dashboard/features.go:111-117`
**Description:** Fetches 100 runs for the repo, then filters by issue number in Go code. Should use SQL `WHERE repo = ? AND issue = ?` for efficiency.

**Fix:** Add a `GetRunsByIssue(repo, issue)` database function.

## Architecture Recommendations

1. **Add authentication**: At minimum, a shared-secret bearer token for admin and internal endpoints. The dashboard currently has zero access control.

2. **Extract shared helpers**: The repo profile loading function is duplicated 4 times. Create a shared `config` package.

3. **Use context-based timeouts consistently**: Many external calls (git clone, AI API, GitHub API) lack proper context cancellation. This causes goroutine leaks on shutdown.

4. **Add database migration versioning**: The current approach of running `ALTER TABLE ... ADD COLUMN` on every startup (ignoring errors) works but is fragile. Consider a proper migration system.

5. **Consider moving large data out of env vars**: The `OPINAI_REPO_CONTEXT` and `OPINAI_DEPLOYMENT_PLAN` env vars can be very large. Consider using K8s ConfigMaps or tmpfs volumes instead.

6. **Add request body size limits**: SSE and chat endpoints accept unbounded request bodies.

7. **Wire `/api/verify-fix` route**: The handler exists but isn't registered.
