## What Is OpinAI

OpinAI is a Kubernetes-native Go service that automates GitHub issue reproduction and pull request review using AI-powered analysis. It runs as a controller pod that polls GitHub repos for new issues, spawns K8s Jobs (runner pods) to investigate them in isolated sandbox namespaces, uses AI to analyze results, and presents findings on a dashboard.

## Two Runtime Modes

- **Controller mode** (`--mode controller`): Long-lived pod that orchestrates everything â€” dashboard server (HTTP :8080 / HTTPS :8443), GitHub poller, job manager, sandbox manager, and database.
- **Runner mode** (`--mode runner`): Short-lived pod created by the controller inside K8s Jobs. Fetches an issue or PR, reproduces/reviews it, runs AI analysis, and reports results back to the controller via HTTP callback.

Entry point: `cmd/opinai/main.go` â€” the `--mode` flag selects which mode to run.

## Key Data Flow

For issues:
```
GitHub Issues -> Poller -> JobManager -> K8s Job (runner pod)
  -> Sandbox namespace -> Reproduce -> AI categorize/verdict -> DB + Dashboard
```

For PR reviews:
```
GitHub Webhook (or manual trigger) -> JobManager -> K8s Job (runner pod)
  -> Clone repo -> Checkout PR branch -> AI review -> DB + Dashboard
```

## Concurrency Model

- Maximum **3 concurrent** reproduction/review jobs at any time
- Only **1 job per repo** at a time (prevents resource contention)
- Excess issues go into a **pending queue** (`pending_reproductions` table)
- When a job completes, pending items for that repo are automatically retried
- PR reviews use a separate pending queue (`pending_pr_reviews` table)

## Database

SQLite with WAL mode, pure Go driver (`modernc.org/sqlite`, no CGO needed). Write operations serialized with a mutex. Default path: `/data/opinai.db` (configurable via `OPINAI_DB_PATH`).

## Issue Investigation Process

1. Poller fetches open issues from monitored repos via GitHub API
2. New issues (not in `processed_issues`) get queued
3. JobManager creates a K8s Job with a runner pod
4. Runner fetches the full issue (title, body, comments, linked PRs)
5. AI classifies: does this need runtime testing or code review?
6. **Runtime testing**: clone repo, build, deploy to sandbox namespace, run tests
7. **Code review**: clone repo, analyze relevant source files with AI
8. AI categorizes (BUG, FEATURE, QUESTION, DOCS) and gives verdict (BUG_CONFIRMED, NOT_REPRODUCIBLE, NOT_A_BUG, FEATURE_REQUEST, etc.)
9. **Self-healing**: if first attempt fails, AI analyzes the failure and retries with fixes
10. Results sent back to controller via HTTP callback (`POST /api/internal/result`), stored in DB, shown on dashboard

## PR Review Flow

1. GitHub webhook receives PR event, OR user triggers manually from dashboard
2. Controller creates a K8s Job with runner in PR review mode
3. Runner clones the repo, checks out the PR branch, analyzes the diff
4. AI reviews code changes, identifies risks, suggests improvements
5. Result saved to `pr_reviews` table and shown on dashboard
6. User can optionally post the review as a GitHub comment â€” **never auto-posts**
7. Verdicts: APPROVE, CHANGES_REQUESTED, or COMMENT
8. Risk levels: LOW, MEDIUM, HIGH, CRITICAL

## Memory System

OpinAI builds up knowledge about each repo over time:

- **repo_memory**: Key-value store â€” language, framework, build commands, runtime requirements, API surface, rich analysis
- **memory_events**: Full audit trail â€” every change logged with old/new value, reason (e.g., "investigation #42"), and source (runner/controller)
- **investigation_findings**: Per-file findings â€” which files were investigated in each issue, what was found, verdict and confidence
- **outcomes**: Accuracy tracking â€” compares OpinAI's verdict against actual outcome (e.g., issue closed after BUG_CONFIRMED = correct prediction)

Memory is built from: deep repo analysis (AI agent reads code), runner investigations, and deployment planning.

## Dashboard Pages

- **Main Dashboard** (`/`): System status, monitored repos, unified issue results table, PR reviews, URL input bar for manual analysis
- **Intelligence** (`/intelligence`): Per-repo investigation findings, file hotspots, outcome tracking, accuracy metrics
- **Memory** (`/memory`): Memory journal, audit trail of repo knowledge changes over time
- **Admin** (`/admin`): System settings, AI/GitHub connectivity testing, deployment plans, sandbox management, infrastructure dependencies, logs

## AI Providers

Three providers supported (set via `AI_PROVIDER` env var):

- **Anthropic** (default): Claude models via direct API. Default model: `claude-sonnet-4-20250514`
- **OpenAI**: GPT models via OpenAI-compatible API
- **Vertex AI**: Claude models via Google Cloud Vertex AI (requires GCP project/region config)

Streaming supported for chat. Single-turn calls used for categorization, verdicts, and analysis.

## Sandbox System

Runner pods get isolated sandbox namespaces with:
- Network policies (restricted egress)
- Resource quotas (CPU/memory limits)
- Automatic cleanup after 30 minutes
- Managed by the sandbox manager in controller mode

## Infrastructure Management

OpinAI can auto-install infrastructure dependencies (PostgreSQL, Redis) into a dedicated `opinai-infra` namespace using Helm. These are used when a repo under investigation needs a database or cache for runtime testing.
