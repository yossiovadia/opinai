## Evaluation Criteria

Score 1-10 against these Distinguished Engineer standards:

### Depth (weight: HIGH)
- Did the review analyze WHY changes matter, not just WHAT changed?
- Did it trace causality? ("This changes X, which means Y will now behave differently under Z conditions")
- A 7+ review identifies second-order effects. A 4- review just paraphrases the diff.

### Missed Angles (weight: HIGH)
- Security: input validation, auth boundaries, injection vectors, secrets handling, permission escalation
- Concurrency: race conditions, lock ordering, goroutine/thread leaks, shared mutable state, channel semantics
- Error propagation: does the error path actually work? Are errors swallowed? Do partial failures leave inconsistent state?
- Backward compatibility: API contracts, serialization format changes, database schema evolution, config key changes
- Operational risks: rollout safety, rollback plan, feature flags, metric/alert gaps, log verbosity changes

### Architectural Awareness (weight: MEDIUM)
- Did the review connect this change to the broader system?
- Did it identify if this change creates or resolves tech debt?
- Did it consider if this pattern should be applied elsewhere (or already exists elsewhere)?
- For controllers/operators: did it consider reconciliation loops, watch predicates, status conditions?

### Cross-Repo / Cross-Service Awareness (weight: MEDIUM)
- Could this change break consumers? API clients? Dependent services?
- Does it affect shared types, proto definitions, or interface contracts?

### Actionable Value (weight: HIGH)
- Would a senior engineer learn something from reading this review?
- Does it teach or just gatekeep?
- Are suggestions specific enough to act on? ("Consider using a sync.Pool here" vs "consider performance")

### False Positives (weight: MEDIUM — deduct points for these)
- Did the review flag things that aren't actually issues?
- Did it nitpick style when there are real problems to find?
- Did it hallucinate issues not present in the code?

### Blind Spots (weight: HIGH — what should it have caught?)
- Given the repo context and existing meta-learnings, what patterns should the reviewer have recognized?
- What questions should a distinguished engineer ask that this review didn't?
- Are there known gotchas in this codebase (from meta-learnings or repo memory) that were ignored?
