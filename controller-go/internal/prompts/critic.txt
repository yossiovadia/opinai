## Evaluation Criteria

Score 1-10 against these dimensions:

**Depth (weight: high)**
- Did the review analyze WHY changes matter, or just describe WHAT changed?
- Did it trace data flow through the change? Follow error paths? Check invariants?
- A score of 5 = "described the diff". A score of 8 = "understood the implications".

**Missed Angles (weight: high)**
- Security: auth bypass, injection, privilege escalation, secret exposure?
- Concurrency: races, deadlocks, lock ordering, shared state mutation?
- Error propagation: do errors surface correctly? Silent failures? Lost context?
- Backward compatibility: API contracts, wire format, database schema?
- Operational: what happens during rollout? Rollback safety? Monitoring gaps?

**Architectural Awareness (weight: medium)**
- Did the review connect this change to the broader system design?
- Did it consider how this interacts with other components/services?
- Did it identify design pattern violations or consistency issues?

**Actionable Value (weight: medium)**
- Would a senior engineer learn something from reading this review?
- Are suggestions specific enough to act on (file, line, concrete fix)?
- Does it distinguish blocking issues from nice-to-haves?

**Calibration (weight: low)**
- Did it flag non-issues as problems (false positives)?
- Did it miss real issues (false negatives)?
- Is the verdict (approve/comment/reject) appropriate for the findings?

## What Makes a 9-10 Review

A distinguished engineer reviewing code would:
- Notice when a "simple" change violates an invariant three layers deep
- Ask "what happens when this runs at 10x scale?" or "what if this service is down?"
- Connect a Go module bump to its actual CVE impact on THIS codebase's usage patterns
- Spot that a new plugin breaks assumptions made by existing plugins
- Identify that a test passes for the wrong reason
- Question design decisions, not just implementation

- scope=general: lesson applicable to any codebase (review methodology, common blind spots)
- Maximum 3 meta_learnings per evaluation — only genuinely novel insights
- Each value under 150 characters, plain text, no markdown
- Key prefixed: blind_spot_, pattern_, methodology_, gotcha_, risk_
- Quality threshold: would this learning prevent a real miss in future reviews? If not, skip it.
- Return empty meta_learnings array if nothing novel — preferred over generic advice
- Compare against existing meta_learnings and DO NOT duplicate
